## 1. Обзор проекта

### 1.1 Описание

**F-Guild** — кроссплатформенная семейная геймифицированная платформа в стиле Dungeons & Dragons, где родители (гейм-мастера) создают квесты-задания для детей (искателей приключений). Платформа превращает повседневные обязанности в увлекательное приключение с полноценной DnD-механикой: классы, способности, броски d20, баффы/дебаффы, торговая гильдия и система молитв.

### 1.2 Проблема

Родители сталкиваются с трудностями при мотивации детей к выполнению домашних обязанностей, учёбе и развитию полезных привычек. Существующие таск-трекеры для детей либо слишком примитивны (простые чеклисты), либо не учитывают семейный контекст. Геймификация через знакомую и популярную DnD-механику создаёт глубокую вовлечённость, систему ценностей (честность через «доблесть», стратегическое мышление через ресурс-менеджмент) и долгосрочную мотивацию.

### 1.3 Видение

Стать ведущей платформой семейной геймификации, которая делает взаимодействие родителей и детей увлекательным, обучающим и эффективным.

### 1.4 Цели MVP

- Полнофункциональная веб-платформа с полным набором игровых механик
- Мультиязычность с коробки (i18n)
- Архитектура, готовая к быстрому развёртыванию мобильных приложений (PWA → нативные)
- Инфраструктура подписочной модели (без активации на MVP)
- Поддержка до 10 000 гильдий на старте
- Оффлайн-режим

## 2. Целевая аудитория

### 2.1 Первичная аудитория

- **Родители** (25–45 лет) с детьми от 10 лет, ищущие способ мотивировать детей
- Семьи, знакомые с концепцией DnD или настольных ролевых игр
- Технически грамотные родители, готовые использовать цифровые инструменты для воспитания

### 2.2 Вторичная аудитория

- Семьи без опыта в DnD, привлечённые геймификацией
- Многодетные семьи (до 10 детей в одной гильдии)
- Семьи с совместной опекой (до 5 гейм-мастеров)

### 2.3 Персоны

**Гейм-мастер (родитель):**
- Хочет структурировать обязанности детей
- Хочет мотивировать через игру, а не через наказания
- Нуждается в контроле и видимости прогресса

**Искатель приключений (ребёнок 10+):**
- Хочет получать награды за выполненные задания
- Любит прокачку персонажа и игровые механики
- Хочет чувствовать выбор и влияние на свою «историю»

## 3. Роли и иерархия пользователей

### 3.1 Три уровня доступа

| Роль | Описание | Возможности |
|------|----------|-------------|
| **Админ платформы** | Команда F-Guild | Управление всеми гильдиями, помощь по обращениям, модерация, автоочистка |
| **Гейм-мастер** | Родитель/опекун | Управление гильдией, квестами, магазином, баффами, просмотр дашборда |
| **Искатель приключений** | Ребёнок (10+) | Выполнение квестов, покупки, использование способностей, молитвы |

### 3.2 Структура гильдии

- Создатель гильдии — **главный мастер** (суперадмин гильдии)
- Может добавить от 1 до 4 дополнительных мастеров (всего до 5)
- Главный мастер может удалять/понижать других мастеров
- Максимум игроков в гильдии: **10**

### 3.3 Аутентификация

**Гейм-мастер (регистрация):**
- Email + пароль
- OAuth 2.0 (Google, Apple)
- JWT token-based session management

**Искатель приключений (создаётся мастером):**
- Вход по уникальному имени (в рамках гильдии) + PIN-код
- Вход по QR-коду (генерируется в дашборде мастера)
- Опционально: email + пароль (для подростков)

**Acceptance criteria:**
- Мастер создаёт аккаунт, верифицирует email, создаёт гильдию
- Мастер добавляет игроков, выбирая метод входа (PIN/QR/email)
- Игрок входит через выбранный метод и видит свой дашборд
- Все сессии persistent across browser restarts
- Password recovery для мастеров через email
- PIN reset через дашборд мастера

## 4. Основные функции

### 4.1 Квестовая система

#### 4.1.1 Создание квестов (Мастер)

**Поля квеста:**

| Поле | Тип | Обязательное | Описание |
|------|-----|-------------|----------|
| `title` | string | Да | Название квеста |
| `description` | text | Да | Описание задания |
| `type` | enum | Да | `mandatory` / `optional` |
| `recurrence` | enum | Нет | `once` / `daily` / `weekly` / `monthly` / `custom` |
| `deadline` | datetime | Нет | Срок выполнения |
| `xp_reward` | integer | Да | Очки опыта за выполнение |
| `gold_reward` | integer | Да | Золото за выполнение |
| `faith_reward` | integer | Нет | Очки веры за выполнение |
| `difficulty` | integer | Да | Класс сложности (DC) для проверок |
| `confirmation_type` | enum | Да | `photo` / `text` / `timer` / `master_confirm` |
| `assigned_to` | array<player_id> | Нет | Конкретные игроки (null = все) |
| `image` | url | Нет | Иллюстрация квеста |

**Типы квестов:**
- **Обязательные (`mandatory`)** — появляются у игрока в списке, нельзя отклонить
- **Опциональные (`optional`)** — публикуются на «доске гильдии», игрок может принять или пропустить

**Повторяющиеся квесты:**
- Создаются один раз, автоматически появляются по расписанию
- Каждый день/неделю/месяц создаётся новый инстанс
- Отображается прогресс-трекер (например, 5/7 дней выполнено на этой неделе)

#### 4.1.2 Выполнение квестов (Игрок)

**Workflow:**
1. Игрок видит квест в своём списке или на доске гильдии
2. Для опциональных — нажимает «Принять квест»
3. Выполняет задание в реальной жизни
4. Подтверждает выполнение (фото/текст/таймер)
5. Мастер получает уведомление о pending подтверждении
6. Мастер принимает / отклоняет / возвращает на доработку
7. При принятии — начисляются XP, золото, очки веры
8. Мастер может наложить бафф/дебафф по результату

**Типы подтверждений:**
- `photo` — загрузка фотографии (например, фото застеленной кровати)
- `text` — текстовое описание выполненного
- `timer` — таймер на выполнение (например, 30 минут чтения)
- `master_confirm` — мастер вручную подтверждает через свой дашборд

**Acceptance criteria:**
- Квест создаётся с полным набором параметров
- Повторяющиеся квесты автоматически генерируют новые инстансы
- Дедлайн отображается с обратным отсчётом
- Фото сохраняется в хранилище, привязывается к квесту
- Мастер может отклонить с комментарием причины
- XP/золото/вера начисляются атомарно при принятии

### 4.2 Система персонажей

#### 4.2.1 Классы

Классический набор DnD. MVP-классы:

| Класс | Основная характеристика | Уникальная механика |
|-------|------------------------|---------------------|
| **Воин (Fighter)** | Сила | Доблесть — признание проступков смягчает наказание (бросок d20) |
| **Маг (Wizard)** | Интеллект | Заклинания — бонусы к учебным квестам |
| **Лучник (Ranger)** | Ловкость | Следопыт — бонусы к квестам на свежем воздухе |
| **Жрец (Cleric)** | Мудрость | Благословение — восстановление очков веры |
| **Плут (Rogue)** | Харизма | Торг — скидки в торговой гильдии |
| **Бард (Bard)** | Харизма | Вдохновение — бонус к броскам другого игрока |

#### 4.2.2 Модель персонажа

| Поле | Тип | Описание |
|------|-----|----------|
| `name` | string | Имя персонажа (уникальное в гильдии) |
| `class` | enum | Выбранный класс |
| `level` | integer | Текущий уровень (начинается с 1) |
| `xp` | integer | Текущий опыт |
| `xp_to_next_level` | integer | Опыт до следующего уровня |
| `gold` | integer | Внутриигровая валюта (стартовые: 100) |
| `faith_points` | integer | Очки веры (стартовые: 10) |
| `abilities` | array<ability> | Открытые способности |
| `inventory` | array<item> | Предметы в инвентаре |
| `active_buffs` | array<buff> | Активные баффы/дебаффы |
| `stats` | object | Характеристики персонажа |

#### 4.2.3 Система уровней

- Уровни с 1 по 20 (классика DnD)
- При повышении уровня: выбор новой способности из дерева класса
- Прогрессивная шкала XP (каждый следующий уровень требует больше опыта)
- Формула: `xp_to_next = base_xp * level * modifier`

#### 4.2.4 Система способностей

- Каждый класс имеет дерево способностей
- При левел-апе игрок выбирает одну новую способность из доступных
- Способности могут требовать предыдущие способности (пререквизиты)
- Активные способности требуют броска d20
- Пассивные способности действуют постоянно

**Acceptance criteria:**
- Игрок выбирает класс при создании персонажа
- Дерево способностей отображается визуально
- При левел-апе доступные способности подсвечиваются
- Использование активных способностей логируется
- Мастер видит все использования способностей в дашборде

### 4.3 Дайс-механика (d20)

#### 4.3.1 Бросок кубика

**Формула проверки (классика DnD):**

```
Результат = бросок d20 + модификатор способности + модификатор предмета + модификатор баффа
Успех = Результат >= Класс Сложности (DC)
```

**Визуализация:**
- Анимированный электронный d20
- Отображение формулы: `[14] + 2 (меч) + 3 (благословление) = 19 vs DC 15 → Успех!`
- Результат сохраняется в логе

**Модификаторы:**
- Предметы из инвентаря (например, «Амулет мудрости +1»)
- Активные баффы/благословления (например, «Благословление отца +3»)
- Классовые способности
- Характеристики персонажа

#### 4.3.2 Прозрачность бросков

- Каждый бросок записывается в лог с timestamp
- Мастер видит в дашборде: кто бросал, что выпало, какие модификаторы, результат
- Бросок нельзя перебросить (если не использована специальная способность)
- Серверная генерация случайного числа (для честности)

**Acceptance criteria:**
- Бросок d20 генерируется на сервере (не клиенте)
- Анимация кубика на клиенте для UX
- Полный лог бросков доступен мастеру
- Модификаторы автоматически подтягиваются из инвентаря и баффов
- Результат определяется по формуле DnD

### 4.4 Баффы и дебаффы

#### 4.4.1 Управление (Мастер)

Отдельный блок в дашборде мастера для создания и управления баффами/дебаффами.

**Модель баффа/дебаффа:**

| Поле | Тип | Описание |
|------|-----|----------|
| `name` | string | Название (например, «Благословление предка») |
| `type` | enum | `buff` / `debuff` |
| `effect` | object | Эффект: `{ type: 'xp_bonus', value: 50 }` |
| `duration` | enum | `permanent` / `timed` / `manual_cancel` |
| `duration_value` | integer | Длительность в минутах (для timed) |
| `description` | text | Описание для игрока |
| `icon` | string | Иконка |

**Типы эффектов:**
- `xp_bonus` / `xp_penalty` — процент к XP за квесты (например, +50%)
- `gold_bonus` / `gold_penalty` — процент к золоту
- `gold_drain` — списание суммы (проклятие рассеянности)
- `dice_bonus` / `dice_penalty` — модификатор к броскам d20
- `shop_discount` / `shop_markup` — изменение цен в магазине
- `faith_bonus` — бонус к очкам веры
- `custom` — произвольный эффект с описанием

#### 4.4.2 Применение

- Мастер накладывает бафф/дебафф на конкретного игрока
- Мастер выбирает длительность или ставит на ручную отмену
- Игрок видит активные эффекты на своём профиле
- Тайминговые эффекты автоматически снимаются по истечении
- Мастер может отменить любой эффект досрочно

**Acceptance criteria:**
- Мастер создаёт шаблоны баффов/дебаффов в отдельном разделе
- Применение к игроку — в один клик из шаблона
- Игрок видит список активных эффектов с таймерами
- Автоматическое снятие по таймеру
- История всех применений в логе

### 4.5 Торговая гильдия

#### 4.5.1 Товары

Мастер создаёт товары двух категорий:

**Реальные награды:**
- «Поход в кино» — 500 золотых
- «Час PlayStation» — 200 золотых
- «Выбор ужина» — 150 золотых

**Игровые предметы:**
- «Меч удачи +2» — даёт модификатор к броскам d20
- «Зелье здоровья» — восстанавливает очки (механика на усмотрение мастера)
- «Индульгенция» — снимает один дебафф
- Предметы с уровневыми и классовыми ограничениями

**Модель товара:**

| Поле | Тип | Описание |
|------|-----|----------|
| `name` | string | Название |
| `description` | text | Описание |
| `category` | enum | `real_reward` / `game_item` |
| `price` | integer | Цена в золоте |
| `stock` | integer / null | Количество (null = бесконечно) |
| `level_required` | integer | Минимальный уровень |
| `class_required` | enum / null | Класс (null = любой) |
| `effect` | object / null | Игровой эффект предмета |
| `image` | url | Изображение |
| `is_active` | boolean | Доступен ли в магазине |

#### 4.5.2 Покупка

1. Игрок просматривает каталог (фильтры: категория, доступные по уровню/классу)
2. Нажимает «Купить»
3. Золото списывается, предмет добавляется в инвентарь
4. Мастер получает уведомление о покупке
5. Для реальных наград — мастер должен подтвердить исполнение

**Acceptance criteria:**
- Мастер создаёт/редактирует/удаляет товары
- Игрок видит только доступные по уровню и классу товары
- Покупка атомарна (золото списывается + предмет добавляется в одной транзакции)
- Мастер видит историю покупок каждого игрока
- Реальные награды требуют подтверждения мастера

### 4.6 Молитвы (Система сообщений)

#### 4.6.1 Механика

- Игрок отправляет свободное текстовое сообщение мастеру
- Каждая молитва стоит **очки веры**
- Стартовый баланс: **10 очков веры**
- Очки веры пополняются: мастером вручную или через квесты

#### 4.6.2 Приватность

- Молитвы — **приватный канал** между игроком и мастерами гильдии
- Другие игроки не видят чужих молитв
- Все мастера гильдии видят молитвы всех игроков

#### 4.6.3 Ответы

- Мастер может отвечать на молитвы
- Формат: threaded-сообщения (молитва → ответ → ответ)

**Модель молитвы:**

| Поле | Тип | Описание |
|------|-----|----------|
| `player_id` | uuid | Отправитель |
| `message` | text | Текст молитвы |
| `faith_cost` | integer | Потраченные очки веры |
| `status` | enum | `sent` / `read` / `answered` |
| `replies` | array<reply> | Ответы мастера |
| `created_at` | datetime | Время отправки |

**Acceptance criteria:**
- Игрок видит свой баланс очков веры
- Отправка молитвы списывает очки веры
- Если очков недостаточно — кнопка отправки неактивна
- Мастер получает уведомление о новой молитве
- Мастер отвечает, игрок видит ответ в тредовом формате

### 4.7 Дашборд мастера

**Главный экран:**
- Обзор всех игроков: уровень, XP, золото, активные квесты
- Ожидающие подтверждения квесты (pending)
- Новые молитвы (непрочитанные)
- Последние действия (activity feed)

**Разделы:**
- **Квесты** — создание, редактирование, просмотр статусов
- **Игроки** — профили, прогресс, баффы, инвентарь каждого
- **Торговая гильдия** — управление товарами
- **Баффы/дебаффы** — шаблоны и применение
- **Молитвы** — входящие сообщения с ответами
- **Дайс-лог** — история всех бросков d20
- **Настройки гильдии** — управление мастерами, настройки уведомлений
- **Настройки профиля** — уведомления (push/email/дашборд), реклама

### 4.8 Дашборд игрока

**Главный экран:**
- Карточка персонажа (аватар, уровень, класс, XP-бар)
- Активные квесты
- Баланс (золото, очки веры)
- Активные баффы/дебаффы с таймерами

**Разделы:**
- **Доска квестов** — обязательные + доступные опциональные
- **Мои квесты** — принятые и в процессе
- **Персонаж** — характеристики, дерево способностей, инвентарь
- **Торговая гильдия** — магазин
- **Молитвы** — отправка и история
- **Кубик** — бросок d20 (когда требуется)

## 5. Рекомендации по техническому стеку

### 5.1 Обоснование выбора

Стек выбран с учётом требований 2026 года: масштабируемость до 10 000+ гильдий, мультиязычность, оффлайн-режим, быстрый переход на мобильные приложения, разработка AI-агентами.

### 5.2 Рекомендуемый стек

| Компонент | Технология | Обоснование |
|-----------|-----------|-------------|
| **Frontend** | Next.js 15+ (App Router) + React 19 | SSR/SSG, i18n из коробки, PWA-ready |
| **Стилизация** | Tailwind CSS 4 + shadcn/ui | Кастомизируемая дизайн-система, тематизация |
| **Backend** | Node.js + tRPC или Hono | Type-safe API, отличная совместимость с Next.js |
| **БД** | PostgreSQL + Prisma ORM | Надёжность, JSONB для гибких данных, отличная экосистема |
| **Кэш** | Redis | Сессии, real-time данные, кэширование |
| **Auth** | NextAuth.js (Auth.js) v5 | OAuth, credentials, JWT, мультипровайдер |
| **Real-time** | WebSocket (Socket.io или Pusher) | Уведомления, обновление дашборда |
| **Хранилище файлов** | Cloudflare R2 или AWS S3 | Фото подтверждений квестов |
| **Email** | Resend | Транзакционные письма, уведомления |
| **Push** | Firebase Cloud Messaging (FCM) | Push-уведомления, кроссплатформенные |
| **i18n** | next-intl | Мультиязычность с коробки |
| **Оффлайн** | Service Workers + IndexedDB | PWA offline-first |
| **Аналитика** | PostHog (self-hosted) или Plausible | Privacy-friendly, GDPR-compliant |
| **Платежи (инфраструктура)** | Stripe | Подписки, billing portal, webhook |
| **Хостинг** | Vercel (frontend) + Railway/Fly.io (backend) | Auto-scaling, edge deployment |
| **CI/CD** | GitHub Actions | Автоматизация, интеграция с AI-агентами |
| **Монорепо** | Turborepo | Организация кода для мульти-пакетной архитектуры |

### 5.3 Архитектура

```
┌─────────────────────────────────────────────────┐
│                   Client Layer                   │
│  Next.js 15 (React 19) + Tailwind + shadcn/ui  │
│  PWA + Service Workers + IndexedDB (offline)     │
├─────────────────────────────────────────────────┤
│                    API Layer                      │
│           tRPC / REST API (Hono)                 │
│         WebSocket (real-time updates)            │
├─────────────────────────────────────────────────┤
│                  Service Layer                   │
│  Auth │ Quests │ Characters │ Shop │ Dice │ Chat │
├─────────────────────────────────────────────────┤
│                   Data Layer                     │
│  PostgreSQL │ Redis │ S3/R2 (media)              │
├─────────────────────────────────────────────────┤
│               External Services                  │
│  Resend │ FCM │ Stripe │ PostHog                 │
└─────────────────────────────────────────────────┘
```

### 5.4 Оффлайн-режим

**Стратегия: Offline-first с синхронизацией**

- Service Worker кэширует app shell и статические ресурсы
- IndexedDB хранит локальное состояние (квесты, персонаж, инвентарь)
- При отсутствии сети: игрок видит свои данные, может отмечать квесты
- При восстановлении — фоновая синхронизация с конфликт-резолюцией
- Конфликты: server-wins для критических данных (XP, золото), client-wins для черновиков

## 8. Мультиязычность (i18n)

### 8.1 Подход

- **Библиотека:** next-intl
- **Формат переводов:** JSON (один файл на язык, на namespace)
- **Определение языка:** Язык браузера → выбор пользователя → сохранение в профиле
- **URL-стратегия:** Prefix-based (`/en/dashboard`, `/ru/dashboard`, `/fr/dashboard`)

### 8.2 Стартовые языки

- Английский (en) — по умолчанию
- Русский (ru)
- Французский (fr)

### 8.3 Требования к коду

- Все строки UI — через ключи переводов, никакого хардкода
- Даты и числа форматируются через Intl API
- Контент мастера (квесты, товары, баффы) — на языке мастера, без автоперевода
- RTL-поддержка заложена в архитектуру на будущее

## 9. Безопасность и приватность

### 9.1 Защита данных детей

- **GDPR-K (ЕС)** и **COPPA (США)** compliance
- Минимальный сбор данных: имя/никнейм, PIN (hashed), опционально email
- Возраст не хранится явно — только подтверждение «10+ лет» от мастера
- Нет сбора геолокации, контактов или иных чувствительных данных
- Фото-подтверждения хранятся с ограниченным сроком хранения
- Согласие родителя (мастера) фиксируется при создании аккаунта игрока

### 9.2 Политика хранения данных

- **Фото квестов:** Автоудаление через 90 дней после подтверждения (настраиваемо)
- **Неактивные гильдии:** Предупреждение на email через 11 месяцев → удаление через 12 месяцев неактивности
- **Удалённые гильдии:** Soft delete → полное удаление данных через 30 дней
- **Логи:** Ротация по настраиваемому расписанию

### 9.3 Безопасность

- Все пароли и PIN-коды — bcrypt-хеширование
- HTTPS everywhere
- Rate limiting на API endpoints
- CSRF-токены
- XSS-защита (sanitization входных данных)
- SQL injection prevention (Prisma ORM parameterized queries)
- JWT с коротким сроком жизни + refresh tokens
- QR-токены — одноразовые с ротацией

### 9.4 Доступ

- Данные гильдии видны только членам гильдии
- Админ платформы имеет доступ только при обращении мастера с Guild ID
- Нет публичных профилей или социальных функций
